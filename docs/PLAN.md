### M13. Minecraft Agent 強化ロードマップ

目的は、現状の「単独 AgentRunner + 低レベル MCP ツール」構成を強化し、実行安全性、進捗管理、Discord 連携を改善すること。
実装は一気に進めず、観察可能性とロールバック容易性を優先して段階的に進める。

### M13a. 要件整理とギャップ分析（完了）

- 現状実装の強みと不足を整理した。
- 主な不足: 実行前検証、長期進捗管理、社会行動、スキル運用、安全機構。
- 詳細は `CHANGELOG.md` を参照。

### M13c. イベント駆動化と実行安全性の強化

- 30 秒ポーリング中心の判断に加えて、危険イベント・被ダメージ・接近 mob・キック・死亡を優先処理する再判断経路を設ける。
- 「平常時の長め判断」と「危険時の短い即応判断」を分離する。
- stop / flee / eat / shelter のような生存行動は、LLM 判断を待たずに走れる条件を明文化する。
- 実行中ジョブのキャンセル条件、再試行回数、クールダウン、Discord 通知条件を整理する。

2026-03-11 時点の整理結果:

- 危険時プリエンプションの標準手順は `stop` → 再観測 → `eat_food` / `flee_from_entity` / `find_shelter` / `sleep_in_bed` から状況依存で次の 1 手を選択する。
- `death`, `kicked`, `disconnect`, `damage`, 近距離 hostile、致命的低体力、空腹 0 を危険時再判断の主要トリガーとする。
- 同系統ジョブが連続 2 回失敗またはキャンセルした場合はクールダウン対象とし、前提条件が変わらない限り即再試行しない。
- 失敗原因は `survival failure`, `pathfinding failure`, `resource shortage`, `target missing`, `connection failure` に分類して扱う。
- Discord 通知は即時通知と要約通知に分け、死亡・切断・依頼失敗・危険回避開始/失敗に加え、危険回避完了・再計画開始・依頼延期・長時間スタックも対象とする。

補強したい機能:

- 危険時プリエンプション
- stuck 検知
- 同じ失敗行動の連続抑止
- inventory 不足時の代替案提示
- 目標未達の理由記録

DoD:

- 危険時に「計画中でも即中断して逃げる」ルールが定義される。
- ツール失敗時の標準ハンドリング方針が決まり、`STATUS.md` の既知リスクとして追える。
- `SPEC.md` と `ARCHITECTURE.md` に危険時プリエンプション、再試行条件、クールダウン、通知条件が反映されている。

### M13e. 正式アカウントログイン対応

目的:

- `auth: "offline"` 前提をやめ、必要に応じて Microsoft アカウントの正式ログインを選べるようにする。
- ローカル検証用の offline 接続と、本番用の正式アカウント接続を共存させる。

検討項目:

- 環境変数設計: `MC_AUTH_MODE`（例: `offline` / `microsoft`）、必要なら `MC_USERNAME` の役割見直し。
- mineflayer の Microsoft 認証フローに必要な資格情報保管方法。
- トークンキャッシュの配置先、失効時の再認証手順、RUNBOOK への反映。
- 2FA / デバイスコード / ブラウザ認証など、常駐プロセス向けに破綻しない運用手順。
- 認証失敗時に bot が無限再接続しないようにする制御。
- viewer や metrics など周辺機能が auth mode 差分で壊れないことの確認。

非目標:

- この段階では複数 Minecraft アカウントの同時運用までは広げない。
- 認証基盤を一般化して Discord や OpenCode 側に波及させない。

DoD:

- offline / microsoft の 2 モードが設定として整理され、秘密情報の置き場が決まる。
- `RUNBOOK.md` に再現可能なログイン手順を記述できる粒度まで設計が固まる。

### M14. LTM（長期記憶）強化ロードマップ

目的は、モノレポに統合した LTM（src/ltm/）を実運用レベルに引き上げること。
現状は「会話を自動でエピソード化し、ファクトを抽出してシステムプロンプトに注入する」基本フローが動いているが、FSRS 学習ループ、記憶の関連性フィルタ、ファイルベースメモリとの責務統合が不足している。

### M14a. FSRS 学習ループの統合

現状: `retrievability()` は検索スコアリングで使用中（fsrsWeight=0.5）だが、`EpisodicMemory.review()` を呼ぶコードが本番に存在しない。stability/difficulty/lastReviewedAt が初期値のまま更新されず、FSRS の「使うほど記憶が強化される」仕組みが機能していない。

対応:

- `ltm_retrieve` 実行時に返されたエピソードを暗黙的に `review()` する（rating は検索ヒット = "good" 相当）
- Consolidation でエピソードが参照された際にも review を記録する
- review 頻度が高いエピソードの retrievability が高くなることをテストで検証する

DoD:

- `ltm_retrieve` 呼び出し後に対象エピソードの lastReviewedAt が更新される
- 複数回 retrieve されたエピソードのスコアが、未 retrieve のエピソードより高い
- `bun test` で FSRS 学習ループの統合テストがパスする

### M14b. ファクト注入の関連性フィルタ

現状: `ContextBuilder.loadLtmFacts()` が `getFacts()` で有効ファクト全件を取得し、システムプロンプトに注入している。ファクト数が増えるとコンテキストウィンドウを圧迫し、関連性の低いファクトがノイズになる。

対応:

- ファクト注入時に Retrieval のハイブリッド検索を使い、直近の会話コンテキストに関連するファクトのみを注入する
- 全件注入から「関連上位 N 件 + カテゴリ別上位」方式に変更する
- `LtmFactReader` インターフェースに `getRelevantFacts(guildId, context, limit)` を追加する

DoD:

- ファクト 100 件以上の環境で、注入されるファクトが関連性でフィルタされる
- システムプロンプトの LTM セクションサイズが制御可能になる

### M14c. 記憶システム責務統合（SPEC Phase 2-3）

現状: SPEC.md §3.6 のマイグレーション Phase 1（ファクト注入）は完了。Phase 2（MEMORY.md スリム化）と Phase 3（日次ログ再設計 + LESSONS.md 整理）が未着手。3 系統の記憶（LTM, MEMORY.md/LESSONS.md, 日次ログ）が独立して運用されており、情報の重複・散逸が発生する。

Phase 2 対応:

- 数日の観察期間を設け、情報ロスがないことを確認する

Phase 3 対応:

DoD:

### M14d. Embedding 堅牢化

現状: embedding モデル（デフォルト: embeddinggemma, 384 次元）の次元数がコード内に定義されておらず、モデル変更時に保存済みデータと新規データの次元がズレてコサイン類似度が壊れる。

対応:

- LtmStorage に embedding 次元メタデータテーブルを追加し、初回保存時に次元を記録する
- 次元不一致を検出した場合にエラーまたは警告を出す
- モデル変更時の再 embedding マイグレーション手順を RUNBOOK.md に記載する

DoD:

- embedding 次元の不一致が検出可能になる
- モデル変更手順が RUNBOOK.md に記載される
