# PLAN.md

## 1. 方針

- OpenCode SDK を AI ランタイムとして使用。責務別フラットモジュール構成（Pure DI）。
- 永続化は SQLite（Drizzle ORM）に統一。MCP サーバーは 4 プロセス構成。
- 詳細なアーキテクチャは `ARCHITECTURE.md` を参照。
- 完了済みマイルストーン（M1-M14d, M13e）の記録は `CHANGELOG.md` を参照。

## 2. 完了定義（DoD）

- `nr validate` が通る
- `bun test` が通る
- ドキュメント（ARCHITECTURE.md, STATUS.md）が現在の実装と一致する
- 手動動作確認: Discord 応答、Heartbeat 自律行動、Minecraft 接続（MC_HOST 設定時）

## 3. 次のマイルストーン

### M13f. Discord 連携の高度化

- Discord からの自然言語依頼を、そのまま丸投げせず「依頼解釈 -> 計画 -> 実行可能タスク」へ変換する。
- 途中経過の共有を「重要時のみ」「うるさすぎない」形に調整する。
- `minecraft_status()` を単なる要約でなく、「現在の目標」「危険度」「実行中タスク」「詰まり理由」込みへ強化する。

DoD:

- ユーザー依頼、Minecraft 自律目標、緊急回避の優先順位が仕様化される。
- Discord 側から見た Minecraft エージェントの説明責任が改善される。

### M13g. 観測性・評価・運用フェーズ

- Minecraft エージェントの評価軸を「生存時間」「依頼完了率」「危険回避率」「目標進捗率」「無駄行動率」で定義する。
- 代表シナリオを固定化して回帰確認できるようにする。

候補シナリオ:

- 夜になったので安全に就寝または退避できる
- 空腹時に食料調達または食事できる
- プレイヤーからの採集依頼を受けて実行状況を返せる
- 危険 mob 接近時に無理な戦闘へ入らない
- 認証失敗時に暴走せず停止・通報できる

DoD:

- 追加すべきメトリクス、ログ、手動確認シナリオが列挙される。
- 「改善したつもり」を避けるための比較基準が決まる。

### M14e. RAG パターン導入（将来検討）

現状: `ltm_retrieve` は MCP ツールとして AI エージェントが明示的に呼ぶ必要がある。新着メッセージに基づいて関連エピソードを自動検索し、コンテキストに注入する RAG パターンが未実装。

### 5.2 残りの改善対象

テストの目的は「壊れた変更を検出し、原因が追いやすく、運用で回し続けられる」こと。
カバレッジ数値を上げること自体は目標にせず、検出力を高める。

1. `DiscordGateway` — discord.js 依存の adapter 層。`isHomeMessage`、`trackEmojiUsage`、`adaptMessage` は private だがロジックを持つ。テスト方針: public API（`onMessage` + `start` で登録した handler）経由で discord.js の `Client` をモックして検証する。ロジックを export するためだけにメソッドを公開しない。
2. `bootstrap.ts` — DI 配線の正しさは統合テストの領域。現時点では深追いしない。

### 5.3 テスト品質の継続運用

- `nr test:quality` の結果を PR ごとに確認する（CI で自動実行済み）
- `nr test:quality:flake` を週次で確認する（CI スケジュール実行済み）
- `lowest_line_coverage_files_top10` から「重要なのにテストがないモジュール」を定期的に拾い上げる
- M13 の各マイルストーンで新規コードを書く際は、対応テストを同時に書く
