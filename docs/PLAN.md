## 3. 次のマイルストーン

### M13. Minecraft Agent 強化ロードマップ

目的は、現状の「単独 AgentRunner + 低レベル MCP ツール」構成を、MindCraft に近い認知分業型へ拡張しつつ、このリポジトリの責務分離と運用性を維持すること。
実装は一気に進めず、観察可能性とロールバック容易性を優先して段階的に進める。

### M13a. 要件整理とギャップ分析

- MindCraft 的な強みを、本プロジェクトに必要な最小単位へ分解して整理する。
- 現状実装の強みと不足を `SPEC.md` / `ARCHITECTURE.md` / `STATUS.md` に反映できる粒度まで言語化する。
- 「会話エージェント」「Minecraft エージェント」「subagent 群」の責務境界を明確化する。

2026-03-11 時点の整理結果:

- 現行の Minecraft エージェントは、単一 `AgentRunner` + 単一プロンプトで観察・計画・実行・報告を兼務している。
- 次段階では、Minecraft エージェント内部をオーケストレータ + `Observer` / `Planner` / `Executor` / `Critic` / `Social` の責務へ分割して設計する。
- subagent は常時並列稼働ではなく、必要時のみ短命起動する。外向き人格の分裂や常駐プロセス増加は M13a の非目標とする。
- 低レベル Minecraft ツールの直接実行権は `Executor` 中心に寄せ、他ロールは要約データと計画データを主に扱う。
- 次の 2〜3 マイルストーンの焦点は `M13b`（認知アーキテクチャ設計）、`M13c`（即応性と安全性）、`M13d`（進捗・記憶再設計）とする。

想定する主な不足機能:

- 認知分業の不足: 現在は 1 つの Minecraft AgentProfile に判断が集中しており、観察・計画・実行・内省が分離されていない。
- 実行前検証の不足: 目標達成に向けた段階計画、前提確認、失敗時の再計画が弱い。
- 長期進捗管理の不足: ワールド進捗、装備段階、拠点状態、探索済み情報の構造化が薄い。
- 社会行動の不足: プレイヤーごとの役割理解、依頼の優先順位付け、報告粒度の最適化が弱い。
- スキル運用の不足: 成功パターン記録はあるが、再利用条件・失敗条件・推奨タイミングまで整理されていない。
- 安全機構の不足: 危険時の強制介入、無限ループ抑止、ツール暴走時のフェイルセーフが不十分。

DoD:

- Minecraft エージェントの今後 2〜3 マイルストーン分の目的、非目的、依存関係が `PLAN.md` と他ドキュメントで矛盾しない。
- subagent 導入理由が「単に賑やかにするため」ではなく、責務分離と品質向上の観点で説明できる。
- `SPEC.md` と `ARCHITECTURE.md` に現行構成、将来責務境界、非目標、主要ギャップが反映されている。

### M13b. Subagent ベースの認知アーキテクチャ設計

方針:

- OpenCode のメイン Minecraft エージェントをオーケストレータとし、その下に責務別 subagent を持つ構成を検討する。
- subagent は常時並列自律ではなく、「必要時に短命で起動される役割プロセス」として設計し、トークン・複雑性を抑える。
- 低レベルツールを全 subagent に直接開放せず、権限を絞る。

候補ロール:

- Observer subagent: `observe_state`, `get_recent_events`, `get_job_status` から状況要約と危険抽出を行う。
- Planner subagent: 直近目標、装備、所持品、周辺状況から数ステップ先までの行動計画を作る。
- Executor subagent: 実行候補のうち 1 手だけを具体ツール呼び出しへ落とす。
- Critic subagent: 実行結果と失敗要因を見て、継続・停止・再計画を判定する。
- Social subagent: Discord から来た依頼や報告要求を整形し、Minecraft 側の優先度へ反映する。

設計論点:

- subagent ごとに使える MCP ツールを制限するか。
- subagent 間通信を OpenCode のコンテキスト内で閉じるか、SQLite / ファイルベースの scratchpad を持つか。
- 1 tick 内で全ロールを毎回回すか、イベント種別ごとに必要ロールだけ起動するか。
- 失敗時に main agent が直接フォールバック実行できるようにするか。

DoD:

- `agent/minecraft/` 配下の責務分割案が定まり、既存 `AgentRunner` と整合する。
- subagent ごとの入力、出力、使用ツール、起動条件、停止条件が文章で定義される。

### M13c. イベント駆動化と実行安全性の強化

- 30 秒ポーリング中心の判断に加えて、危険イベント・被ダメージ・接近 mob・キック・死亡を優先処理する再判断経路を設ける。
- 「平常時の長め判断」と「危険時の短い即応判断」を分離する。
- stop / flee / eat / shelter のような生存行動は、LLM 判断を待たずに走れる条件を明文化する。
- 実行中ジョブのキャンセル条件、再試行回数、クールダウン、Discord 通知条件を整理する。

2026-03-11 時点の整理結果:

- 危険時プリエンプションの標準手順は `stop` → 再観測 → `eat_food` / `flee_from_entity` / `find_shelter` / `sleep_in_bed` の 1 手選択とする。
- `death`, `kicked`, `disconnect`, `damage`, 近距離 hostile、致命的低体力、空腹 0 を危険時再判断の主要トリガーとする。
- 同系統ジョブが連続 2 回失敗またはキャンセルした場合はクールダウン対象とし、前提条件が変わらない限り即再試行しない。
- 失敗原因は `survival failure`, `pathfinding failure`, `resource shortage`, `target missing`, `connection failure` に分類して扱う。
- Discord 通知は即時通知と要約通知に分け、死亡・切断・依頼失敗・危険回避開始/失敗を優先通知対象とする。

補強したい機能:

- 危険時プリエンプション
- stuck 検知
- 同じ失敗行動の連続抑止
- inventory 不足時の代替案提示
- 目標未達の理由記録

DoD:

- 危険時に「計画中でも即中断して逃げる」ルールが定義される。
- ツール失敗時の標準ハンドリング方針が決まり、`STATUS.md` の既知リスクとして追える。
- `SPEC.md` と `ARCHITECTURE.md` に危険時プリエンプション、再試行条件、クールダウン、通知条件が反映されている。

### M13d. ワールド進捗・スキル・記憶の再設計

- `MINECRAFT-GOALS.md` と `MINECRAFT-SKILLS.md` に加え、ワールド状態の構造化メモを持つか検討する。
- 目標を「今やること」だけでなく、技術段階、拠点整備、資源確保、安全保障、依頼対応に分けて管理する。
- スキル記録は「有効条件」「前提装備」「失敗パターン」「再利用可否」を含める。
- Discord からの依頼と自律目標が衝突した場合の優先順位を定義する。

候補ファイル:

- `MINECRAFT-PROGRESS.md`: 装備段階、拠点、農業、探索範囲、主要資源の進捗。
- `MINECRAFT-SOCIAL.md`: プレイヤーごとの依頼、関係性、禁止事項、合意事項。

DoD:

- Minecraft 記憶ファイルそれぞれの責務が重複しない。
- 「何を LLM の文脈へ常時注入するか」と「必要時だけ読むか」の区別が決まる。

### M13e. 正式アカウントログイン対応

目的:

- `auth: "offline"` 前提をやめ、必要に応じて Microsoft アカウントの正式ログインを選べるようにする。
- ローカル検証用の offline 接続と、本番用の正式アカウント接続を共存させる。

検討項目:

- 環境変数設計: `MC_AUTH_MODE`（例: `offline` / `microsoft`）、必要なら `MC_USERNAME` の役割見直し。
- mineflayer の Microsoft 認証フローに必要な資格情報保管方法。
- トークンキャッシュの配置先、失効時の再認証手順、RUNBOOK への反映。
- 2FA / デバイスコード / ブラウザ認証など、常駐プロセス向けに破綻しない運用手順。
- 認証失敗時に bot が無限再接続しないようにする制御。
- viewer や metrics など周辺機能が auth mode 差分で壊れないことの確認。

非目標:

- この段階では複数 Minecraft アカウントの同時運用までは広げない。
- 認証基盤を一般化して Discord や OpenCode 側に波及させない。

DoD:

- offline / microsoft の 2 モードが設定として整理され、秘密情報の置き場が決まる。
- `RUNBOOK.md` に再現可能なログイン手順を記述できる粒度まで設計が固まる。

### M13f. Discord 連携の高度化

- Discord からの自然言語依頼を、そのまま丸投げせず「依頼解釈 -> 計画 -> 実行可能タスク」へ変換する。
- 途中経過の共有を「重要時のみ」「うるさすぎない」形に調整する。
- `minecraft_status()` を単なる要約でなく、「現在の目標」「危険度」「実行中タスク」「詰まり理由」込みへ強化する。
- 必要なら Discord 側にも Minecraft 専用 subagent 的な解釈層を置き、会話人格と作業人格を分離する。

DoD:

- ユーザー依頼、Minecraft 自律目標、緊急回避の優先順位が仕様化される。
- Discord 側から見た Minecraft エージェントの説明責任が改善される。

### M13g. 観測性・評価・運用フェーズ

- Minecraft エージェントの評価軸を「生存時間」「依頼完了率」「危険回避率」「目標進捗率」「無駄行動率」で定義する。
- subagent 導入でトークン消費と遅延が増えるため、メトリクス追加方針を先に決める。
- 代表シナリオを固定化して回帰確認できるようにする。

候補シナリオ:

- 夜になったので安全に就寝または退避できる
- 空腹時に食料調達または食事できる
- プレイヤーからの採集依頼を受けて実行状況を返せる
- 危険 mob 接近時に無理な戦闘へ入らない
- 認証失敗時に暴走せず停止・通報できる

DoD:

- 追加すべきメトリクス、ログ、手動確認シナリオが列挙される。
- 「改善したつもり」を避けるための比較基準が決まる。
