# Vicissitude

## プロジェクト概要

Discord bot「ふあ(hua)」のプロジェクト。Bun, TypeScript, OpenCode で動作する。

## ドキュメント導線

1. `README.md`: 要件定義・仕様

概要や設計の把握のための依存関係グラフ

- `docs/DEPS.md`: プロジェクト全体の依存関係グラフ
- `src/*/DEPS.md`: 各モジュール内の依存関係グラフ
- ※ いずれも commit 時の pre-commit フックで自動生成（手動実行不要）

## コマンド

- `nr lint:fix` — Lint 自動修正 (`oxlint --fix`)
- `nr fmt` — フォーマット (`oxfmt`)
- `nr validate` — fmt:check + lint + check を一括実行
- `nr test` — 全テスト実行
- `nr test:spec` — 仕様テスト (`*.spec.ts`) のみ実行
- `nr test:unit` — ユニットテスト (`*.test.ts`) のみ実行
- `nr deploy` — コンテナを再作成してデプロイ（install → build → 起動）

## デプロイ

- デプロイ前に `hostname` でホストが `Cipher` であることを確認する。`Cipher` 以外ではデプロイしない。
- ソースコード・依存変更後: `nr deploy` で再デプロイ

## 計画フェーズ（Subagent）

ユーザーから新しいタスク・機能要求・バグ報告を受けたとき、いきなり実行に入らず、まず組み込みの `Plan` サブエージェントを起動する。

`Plan` サブエージェントの責務は以下の2つだけ

1. **タスク分解** — 何をどの順番でやるか
2. **スコープ特定** — どのファイル・仕様が関係するか

設計判断（インターフェースをどう切るか）やリスク判断（既存のどこが壊れそうか）は `Plan` サブエージェントの責務外。Plan の出力を受けて、Team Lead が以下のチームメイトに委譲する:

- **設計判断** → `spec-agent`（チームメイト）
- **リスク判断** → `verify-agent`（チームメイト）

## 実行フェーズ（Agent Teams）

4つのチームメイトを agent teams 機能で運用する。各エージェントは対応するスキル（`.claude/skills/` 配下）を使用する。

| チームメイト   | スキル   | 役割                                                       |
| -------------- | -------- | ---------------------------------------------------------- |
| `spec-agent`   | `spec`   | 仕様策定（ユーザー対話）、IF・型設計、仕様テスト、仕様書メンテ |
| `impl-agent`   | `impl`   | 仕様を忠実にコードへ落とし込む                             |
| `test-agent`   | `test`   | 実装コードに対するユニットテスト作成                       |
| `verify-agent` | `verify` | テスト実行・型チェック・lint・仕様整合性の検証、リスク判断 |

### タスク別の呼び出しパターン

各パターンの前に、必ず `plan-agent`（subagent）でタスク分解・スコープ特定を行う。
また、以下の用に agent team を用いて作業を行なう。

**新機能追加:**

1. `spec-agent` → 設計判断 + 仕様テストを書く
2. `verify-agent` → リスク判断
3. `impl-agent` → spec だけ渡して実装させる
4. `test-agent` → unit テストを書く
5. `verify-agent` → 全テスト通るか確認

**バグ修正:**

1. `spec-agent` → 再現テスト（spec）を書く
2. `impl-agent` → 再現テストだけ渡して修正させる
3. `verify-agent` → 既存 spec が壊れてないか確認

**リファクタ:**

1. `verify-agent` → 既存 spec を確認 + リスク判断
2. `impl-agent` → spec だけ渡してリファクタさせる（unit テストは渡さない）
3. `verify-agent` → spec が通るか確認

> リファクタ時に unit テストを渡さないのがポイント。渡すと実装詳細に縛られる。

### テストファイルの分類

仕様テストとユニットテストはファイル拡張子で区別する。

| 種別           | ファイル名  | 書く人     | 目的                                                                         |
| -------------- | ----------- | ---------- | ---------------------------------------------------------------------------- |
| 仕様テスト     | `*.spec.ts` | spec-agent | 公開APIの振る舞い・契約を検証。実装詳細に依存しない（ブラックボックス）      |
| ユニットテスト | `*.test.ts` | test-agent | 内部ロジック・分岐・エッジケースを検証。実装に密結合でOK（ホワイトボックス） |

**配置ルール**:

- 仕様テスト (`*.spec.ts`): トップレベルの `spec/` ディレクトリに、`src/` のディレクトリ構造をミラーして配置する。
- ユニットテスト (`*.test.ts`): ソースファイルと同じディレクトリに co-locate する。

```
spec/agent/
  session-store.spec.ts     # 仕様テスト（公開API契約）

src/agent/
  session-store.ts          # 実装
  session-store.test.ts     # ユニットテスト（内部ロジック詳細）
```

**境界ルール**:

| 観点                 | `*.spec.ts`                  | `*.test.ts`                    |
| -------------------- | ---------------------------- | ------------------------------ |
| テスト対象           | 公開インターフェース・ポート | 内部関数・プライベートロジック |
| モック対象           | 外部依存（DB, API等）のみ    | 内部依存もモック可             |
| リファクタで壊れるか | 壊れてはいけない             | 壊れてよい                     |
| 書くタイミング       | 実装前（TDD）                | 実装後                         |

### Agent Team におけるチームメイトの情報境界

| チームメイト   | 渡すもの                               | 渡さないもの                    |
| -------------- | -------------------------------------- | ------------------------------- |
| `spec-agent`   | タスク説明・既存 `*.spec.ts`・plan出力 | 実装コード・`*.test.ts`         |
| `impl-agent`   | `*.spec.ts`・型定義・SCOPE             | `*.test.ts`・他モジュールの実装 |
| `test-agent`   | 実装コード・`*.spec.ts`                | 他モジュール                    |
| `verify-agent` | `*.spec.ts` のパス・実行結果・plan出力 | 実装の中身                      |

### Team Lead（メインエージェント）の委譲ルール

- Team Lead はチームメイト専用スキル（`spec`, `impl`, `test`, `verify`）を **自分で直接使用してはならない**。これらのスキルは必ず対応するチームメイトに委譲して実行させること。
- 「`spec-agent` で○○をします」と宣言したら、実際に `spec-agent` をチームメイトとして起動すること。自分で spec スキルを呼び出して代行しない。
- Team Lead の責務は **計画・委譲・統合** であり、チームメイトの作業を自ら代行することではない。

## Issue 運用ルール

- 作業中に課題・改善点・技術的負債を発見したら、その場で GitHub Issue を立てる。後回しにしない。
- `/review` 時に指摘した項目のうち、今回の PR で修正しなかったものは必ず Issue に残す。
- Issue はまとめすぎず、トピックごとに小分けにする（1 Issue = 1 トピック）。
- 重要な判断が必要な課題（設計方針の決定、破壊的変更の検討など）には `help wanted` ラベルを付ける。

## Git 戦略

- 作業完了後の PR 作成は確認不要。push してそのまま PR を作成する。
- マージ時はスカッシュマージし、マージ後にリモートブランチを削除する: `gh pr merge <number> --squash --delete-branch`

## 完了宣言ルール

- 「完了」「修正済み」「テスト通過」を主張するには、**そのメッセージ内で実行した検証証拠が必須**
- 前回の実行結果や「通るはず」は証拠にならない
- impl-agent は実装報告時に `nr check` / `nr lint` の実行結果を含めること
- verify-agent の検証は省略不可（「簡単な変更だから不要」は禁止）
