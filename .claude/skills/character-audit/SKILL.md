このホストで動作している bot インスタンスの実運用データを分析し、キャラクター一貫性・記憶品質・ペルソナ定義の改善点を検出して GitHub Issue として報告するスキル。評価対象の bot 名は `extract-audit-data.ts` の出力に含まれる `botName` フィールドで自動判別される。

## Phase 0: データ収集

1. `bun scripts/extract-audit-data.ts > /tmp/character-audit-data.json` を実行してデータをファイルに保存する（stdout が大きいため直接読むと途切れる）
2. `/tmp/character-audit-data.json` を Read で読み込む。ファイルが大きい場合は namespace ごとに分割して読む
3. 以下のファイルを読み込む:
   - `README.md` — プロジェクトコンセプト（目的・要件）
   - `context/SOUL.md` — ペルソナ定義（性格・会話ルール・口調）
   - `context/IDENTITY.md` — 基本情報
   - `context/DISCORD.md` — Discord 応答ルール
4. エピソードが 0 件なら「評価対象なし」として終了

### 複数エージェントの判別

同じ guild に複数の bot インスタンスが参加している場合がある（例: ふあとヤチヨ）。抽出データのトップレベルにある `botName` フィールドが、このホストで動作している bot の名前を示す。

- `botName` と一致する `name` の `role: "assistant"` メッセージが **評価対象**
- それ以外の `role: "assistant"` メッセージは **別インスタンスの発話なので評価対象外**。ただし、会話文脈として参照はする（評価対象 bot の応答が他エージェントの発話を踏まえているかの判断に使う）

guild ごとに overlay context（`LESSONS.md`, `MEMORY.md`）が異なる場合があるので、guild 単位で評価を行う。

## Phase 1: 会話品質評価

各 guild のエピソードを通して、以下の観点で評価する。

### 1.1 ペルソナ一貫性（SOUL.md 照合）

SOUL.md に定義された以下の要素と実際の会話を照合する:

- **性格の核との整合**: SOUL.md の「性格の核」セクションに列挙された特性が維持されているか
- **食いつき/引きの適切さ**: 「食いつくもの」に食いついているか、「引くもの」にはちゃんと反応が変わっているか
- **禁止パターンの違反**: 「会話のルール」「嫌いな口調」に列挙されたパターンに違反していないか

### 1.2 口調・文体（SOUL.md + DISCORD.md 照合）

- SOUL.md の「口調」セクションで定義された一人称・語尾・表現が一貫しているか
- DISCORD.md の「文体・表記」ルール（句読点・絵文字・メッセージ長等）を守れているか
- guild 固有の LESSONS.md に文体ルールがある場合、それも照合する

### 1.3 コンセプト整合性（README.md 照合）

- 「そこで生きているかような自然な存在」として振る舞えているか
- 感情表現・空気読み・無視の判断が適切か
- 全部のメッセージに反応していないか（うざくなるパターン）

### 1.4 LESSONS.md の遵守

guild 固有の `LESSONS.md` がある場合、そこに記載された教訓を守れているか確認する。

## Phase 2: 記憶レイヤー評価

### 2.1 ファクトの品質

- **冗長・重複**: 同じ内容の fact が複数存在しないか
- **過度に一般的**: あまりにも汎用的で行動指針にならない fact がないか
- **古い/矛盾**: 文脈から明らかに古くなっている fact がないか
- **カテゴリの適切さ**: fact の category（identity, preference, guideline 等）が内容と合っているか

### 2.2 記憶の欠落

エピソードの会話内容を見て、fact として保存されるべきだったのに存在しない情報がないか検出する。例:

- ユーザーが明示的に好み・属性・予定を共有した
- 重要な約束や決定が行われた
- ユーザーから「覚えて」と明示的に頼まれた

### 2.3 Consolidation の妥当性

- 未 consolidate のエピソードが大量に溜まっていないか（`consolidatedAt: null`）
- エピソードの要約（summary）が実際のメッセージ内容を適切に反映しているか

## Phase 3: ペルソナ定義の改善示唆

Phase 1・2 の評価を踏まえ、SOUL.md 自体の改善余地を検出する。

- **カバーされていない状況**: 実際の会話で頻出するがSOUL.md に指針がないパターン
- **矛盾する指示**: SOUL.md 内の異なるルールが実際の会話で衝突しているケース
- **効果が薄いルール**: 書かれているが実際に守られていない（= プロンプトとして機能していない）ルール
- **欠けている口調パターン**: 実際にうまくいっている応答の中で、SOUL.md のセリフ例にない良いパターン

## Phase 4: Issue 起票

### 起票基準

以下に該当する問題を Issue にする:

- **繰り返し発生**: 単発ではなく、2つ以上のエピソードで同じ問題が見られる
- **コンセプトに関わる**: 「自然な存在」というコンセプトを損なうレベルの問題
- **原因が特定可能**: どのファイル/設定/ロジックが原因かを指摘できる

単発の些細なブレや、会話の流れで仕方ないものは Issue にしない。

### Issue のスコープ

- 同一の問題が複数 guild で発生している場合は **1つの Issue にまとめる**（guild ID を列挙）
- guild 固有の問題は guild 単位で Issue を作成する

### 重複チェック

起票前に `gh issue list --search "[character-audit]" --state open --limit 50 --json number,title` で既存 Issue を確認し、同じ問題の Issue が既にあればスキップする。

### Issue フォーマット

```
gh issue create \
  --title "[character-audit] <問題の簡潔な説明>" \
  --label "help wanted" \
  --body "$(cat <<'BODY'
## 検出された問題

<問題の説明>

## 証拠

<該当するエピソードからの会話抜粋（guild ID 付き）>

### エピソード 1: <title>
> <user>: <message>
> <botName>: <message>  ← ここが問題

（必要に応じて複数エピソードを列挙）

## 原因分析

- **原因箇所**: <ファイルパスや設定項目>
- **原因の種類**: プロンプト不足 / プロンプト矛盾 / 記憶の問題 / consolidation の問題 / コードの問題
- **詳細**: <なぜこの問題が起きているかの分析>

## 改善の方向性

<具体的な修正案ではなく、方向性を示す>

---
*character-audit スキルによる自動検出*
BODY
)"
```

## Phase 5: 報告

実行結果を以下のフォーマットでログ出力する:

```markdown
## Character Audit 結果

### 評価対象

- 期間: <開始日> 〜 <終了日>
- Guild 数: <N>
- エピソード数: <N>
- ファクト数: <N>

### 検出された問題

- [重要度: high/medium/low] <問題の概要>
  - Guild: <guildId>
  - 原因: <原因箇所>

### 起票した Issue

- #<number>: <title>

### スキップした問題（既存 Issue あり）

- <問題の概要> → 既存 Issue #<number>

### 特記事項

<SOUL.md 改善の示唆、記憶レイヤーの全体的な傾向など>
```

## 制約

- コードの修正は行わない。問題の検出と Issue 起票のみ
- SOUL.md やコンテキストファイルの直接編集は行わない
- `botName` 以外のエージェントの発話は評価対象外
- 個人情報や機密情報を Issue 本文に含めない。Discord 表示名やハンドルネームは公開情報として引用可。会話の抜粋は問題の証拠として必要最小限にとどめる
- 1 回の実行で起票する Issue は最大 3 件まで（重要度が高い順に絞る）
